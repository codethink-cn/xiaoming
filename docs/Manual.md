## 启动小明
### 使用小明之前的准备
#### 小明启动器（Launcher）
小明是一个机器人框架，需要使用启动器启动。启动器一般称为 `Launcher` 我们提供一个简单易用的启动器，名为 `xiaoming-host`，可以在[这里](https://github.com/Chuanwise/xiaoming-bot/releases)下载到最新版本。

#### Java 环境
`xiaoming-host` 是使用 `Java` 编写的，你需要具备 `Java 运行时` 才能启动该程序。如果你的设备尚没有 `Java` 环境，可以到[官网](https://www.java.com/zh-CN/)下载。

#### 一个文件夹
小明初次启动会创建一些文件夹。请不要直接将启动器放在桌面等处，最好专门将其放在一个文件夹中（下文称「小明根目录」或「小明目录」）。

### 编写启动脚本
直接双击小明启动器是可以启动小明的，但现阶段启动器尚未配置 GUI 界面，所以为了不错过控制台输入内容，可以使用编写启动脚本的方式启动它。

请在小明目录下创建文件 `start.bat`（这个名字随意，可以先创建记事本再修改扩展名为 `.bat`），并输入以下内容保存后双击启动：
```bash
java -jar xiaoming-host-xxx.jar
```

### 滑块验证
若是初次在当前设备上登录，则涉及滑块验证等问题。请在 `mirai` 运行时中添加 `JVM` 属性 `mirai.slider.captcha.supported` (添加参数 `-Dmirai.slider.captcha.supported`) ，例如：
```bash
java -Dmirai.slider.captcha.supported -jar xiaoming-host-xxx.jar
```
之后重新启动该脚本。下载[滑块验证助手](https://github.com/mzdluo123/TxCaptchaHelper)，将 `mirai` 显示的弹框内容复制到滑块验证助手中，再将获得的 `ticket` 复制回 `mirai` 显示的弹框后关闭即可。

如果上述方式仍无法正确通过滑块验证，请查阅[最新的 Mirai 滑块验证模块文档](https://github.com/project-mirai/mirai-login-solver-selenium)。

## 初次邂逅
### 给自己的 `QQ` 号授权
最开始，小明并不知道谁是它的主人，任何人都没有权限指挥小明（甚至连查看指令格式的权限都没有）。这个时候需要教小明知道谁是主人。

在控制台（运行启动脚本后看到的黑色界面）处输入下列指令给自己的 QQ 号授权：
```xiaoming
grant <QQ> *
```
在控制台输入的指令和在 QQ 群、临时会话或者私聊的指令并没有什么两样，所以其实使用 `授权 <QQ> *` 也可以，由于编码问题，小明可能无法识别控制台的中文，所以下文介绍控制台指令时，只给出英文版本的指令。

### 带小明前往需要响应的群
将小明邀请进入需要响应的群后，在群内执行下面的指令将本群设置为「响应群」。
```xiaoming
本群启动小明
```
有关响应群的更多细节，可以查看[响应群](#响应群和群标记)。

### 找一个群用来存放小明的日志
小明遇到一些情况时，需要及时向主人反馈消息。我们通过在相应的响应群中执行下面的指令，告诉小明未来将日志发送到这些地方：
```xiaoming
标记本群  log
```
这个指令可以在多个群执行，到时候消息会往这些地方都发一遍。建议只标记一个群为日志群。有关群标记的更多细节，可以查看[群标记](#响应群和群标记)。

## 文件夹
小明运行会产生大量数据，需要保存的部分会被组织在下列文件夹中。

### 插件：plugins
该文件夹内会存在若干 `jar` 文件，称为「插件本体」。与插件同名的文件夹，称为该插件的「数据文件夹」或「插件文件夹」。插件文件夹内会有什么内容，取决于对应的插件。

如果需要新装插件，将插件 `jar` 文件放在该文件夹内重启小明，或执行下列小明指令：
```xiaoming
刷新插件
加载插件 <插件名>
```
看到输出「插件 <插件名> 启动成功」即为插件成功启动。

### 配置信息：configurations
#### 小明基础设置：configurations.json
主要是有关调用限制、明确调用前缀、自动保存和优化周期等非常基础的设置。

|设置项|类型|含义|说明|
|---|---|---|---|
|`debug`|`boolean`|是否开启调试模式|默认关闭|
|`groupCallConfig`|对象|群聊小明调用限制||
|`privateCallConfig`|对象|私聊小明调用限制||
|`maxIterateTime`|`int`|替换消息值得最大次数|将在下文介绍|
|`enableLicense`|`boolean`|是否启动小明使用协议|默认关闭。如果开启，用户首次使用小明时小明会询问是否同意一个协议（内容自定），只有同意了的用户才能和机器人交互。
|`enableClearCall`|`boolean`|是否启动明确调用。明确调用就是规定小明需要响应的信息的开头，小明未来只会响应这些字符串开头的信息|
|`callPrefixs`|集合|只有`enableClearCall`为`true`时才生效|
|`maxUserInputWaitTime`|`long`|在等待用户的下一次输入时，默认最长等待多久就放弃|
|`maxUserGroupInputWaitTime`|`long`|在等待用户的下一次群聊输入时，默认最长等待多久就放弃|
|`maxUserPrivateInputWaitTime`|`long`|在等待用户的下一次私聊输入时，默认最长等待多久就放弃|
|`maxUserGlobalInputWaitTime`|`long`|在等待用户的下一次全局输入时，默认最长等待多久就放弃|
|`optimizePeriod`|`long`|自动优化自身性能的周期
|`savePeriod`|`long`|自动保存需要保存的文件的周期

#### 权限设置：permissions.json
阅读权限文件，你需要一些前置知识。如果你尚未了解相关知识，请阅读下一部分「前置知识」。

##### 权限类别
小明的权限类别按作用域可分「全局权限」和「群权限」

##### 前置知识
小明通过一串文字表示做某件事情的权限。例如，`admin.group.mute.me` 可以表示一个成员禁言自己的权限。只要一个人「具有」这个字符串，就判定其具有该权限。像这样的字符串，就是「权限节点」。

我们可以通过给用户分配权限节点，但是实际上会有很多用户的权限是完全一样的。为了避免无必要的多次分配，我们将一套权限打包为一个「权限组」，在让一些用户属于某个权限组，这样只需要修改权限组，就可以修改该组所有用户的权限。

在所有的权限组中，名为 `default` 的组是最为特殊的。任何新用户都自动属于该组，且如果小明发现不存在默认组，则会自动创建一个不带任何权限，名为 `default` 的组作为默认组。

接下来，我们只需要为权限组分配权限就可以了。

我们很快就会遇到一个新问题，就是有的权限组之间有高低之分。例如「普通用户」和「管理员」，管理员具有用户的所有权限。因此，我们让管理员的权限「继承自」普通用户，就可以让管理员自动拥有普通用户的那些权限。

这是一个权限组的信息：
```json
"test": {
    "superGroups": [
        "java.util.ArrayList",
        [
            "default",
            "testSuper"
        ]
    ],
    "alias": null,
    "permissions": [
        "java.util.ArrayList",
        [
            "-group.tag.add"
        ]
    ],
    "groupPermissions": {
        "@class": "java.util.HashMap"
    }
}
```
`superGroups` 描述了该权限组继承了哪些权限组的权限。`permissions` 是该权限组的全局权限，`groupPermissions` 是该权限组在特定群中的权限。
